== 2026-05-08 23:39 == feat(streak): Streak-Hunt slot pool ==

Fixes a bug where the bot lost the 450-point WATCH_STREAK bonus on
freshly-online channels because all 2 Spade slots were occupied with
rotation. Logs from the past week showed worst-case 1h51min delay
between "went LIVE" and first "Started watching" (t0b1_1337 04.05) —
far past Twitch's streak window.

Mechanic:
- channels.State gains StreakClaimedAt + MarkStreakClaimed setter.
  Survives SetOffline intentionally — the comparison
  StreakClaimedAt < OnlineSince makes a restarted stream re-eligible
  without an explicit reset.
- Rotate() and FillSpadeSlots() honor a new bucket between P0 (drops)
  and P1 (always-watch): online, not drops-owned, StreakClaimedAt
  before OnlineSince, age <30min. FIFO by OnlineSince ASC — oldest
  stream first since its window is closest to expiring.
- farmer.go hooks ReasonCode==WATCH_STREAK -> MarkStreakClaimed and
  immediately calls FillSpadeSlots so the next candidate gets the
  slot within seconds, not 5min.
- Drops keep precedence (P0 > PS) since drops typically pay more
  than 450 points.
- 30min hard timeout prevents one stuck candidate from starving the
  slot pool — scales to 80-90 channels.

== 2026-04-30 12:02 == CLI: --add-channel validates + stores ID; --remove-channel added ==

The rename-resilience logic in addChannelFromEntry has always
worked correctly for channels with an ID stored — it looks up by
ID and silently updates the login if Twitch returns a different
one. Problem: legacy entries with login-only (added via the old
--add-channel CLI flag, or hand-edited into config.json before
ID-tracking was implemented) had no ID to fall back on; once the
streamer renamed/deleted, the entry was unrecoverable.

Fixes:

1. --add-channel now validates the login exists on Twitch via
   GetChannelInfo before persisting, and stores BOTH the login
   AND the resolved channel ID. Future startups can recover
   from renames without user intervention.

2. New --remove-channel flag for cleaning up orphan entries that
   never managed to store an ID before going stale (legacy from
   pre-ID-tracking versions, or hand-edited).

3. Startup error message for orphan entries now hints at
   --remove-channel:
     channel "X" not found on Twitch and no ID stored to recover
     from a rename — remove via `--remove-channel X`

The runtime AddChannelLive (TUI 'a' key + Web Add Channel modal)
was already correct on this — both validate via GetChannelInfo
before persisting, so any channel added via the running bot
already has its ID stored. Only the CLI flag was inconsistent.

== 2026-04-30 01:02 == Fix: synchronous AutoClaim + gameEventDrops fallback ==

   reporting self.isClaimed=false for the same drop for seconds to

   updated immediately and is the reliable signal — but v1.8.0 had
   removed this fallback because the over-eager version broke
   daily-rolling campaigns (Marble Day reuses the same benefit ID
   across days).

- AutoClaimAndMarkCompleted now claims synchronously, mutates the
  in-flight campaigns slice via index loops, sets d.IsClaimed=true
  on success. Downstream stages see the fresh claim without another
  inventory round-trip. Matches TDM (src/models/drop.py:149).
- GetDropsInventory restores the gameEventDrops merge step but
  scopes it to the drop's [StartAt, EndAt) window. Yesterday's
  Marble Day claim doesn't satisfy today's window, so the daily-
  rolling false-positive that drove the v1.8.0 removal stays gone.

gameEventDrops cross-check via the new debugdrops/abi tool showed
the "Infinite Memory Skin" benefit was permanently recorded by

== 2026-04-29 23:57 == Fix: SpadeTracker reverts to legacy POST for channel-points credit ==

Twitch has TWO separate credit pipelines that verify differently:

  Drop-minute credit  → sendSpadeEvents GraphQL mutation (game_id +
                        INT user_id required). Strict anti-cheat.
  Channel-points WATCH credit → legacy POST to spade.twitch.tv/track
                        (or beacon.twitch.tv/track) with the lighter
                        payload (player: "site" / location: "channel").

The v2.0 ABI fix (commit 2cf90a3, earlier today) collapsed both onto
sendSpadeEvents because the comment in the old code said "POST still
returns 204 but no longer credits drops". That was true for drops —
but the inverse half was missed: the GQL mutation does NOT credit
WATCH points. After the switch, drops kept crediting reliably (we
verified extensively for ABI Partner-Only Drops) but channel-points
WATCH credits went silent across all Spade-tracked channels.

Symptom: 484 WATCH credit events from 00:08 to 17:53 today (running
the old code), then 0 WATCH credits for ~6 hours after the bot was
rebuilt with the new code. Drops still fine throughout. The user
opened a real Twitch web tab on cpt_blackshark at 23:51 and credits
resumed within seconds (web session uses POST under the hood) —
proving the account is healthy and the bot's GQL-only path was the
gap.

Fix:
- Restore the legacy POST in internal/twitch/spade.go's sendHeartbeat.
  Uses the pre-2cf90a3 payload (player/location/hidden/live/logged_in/
  muted) and posts as application/x-www-form-urlencoded with the
  Dalvik UA + Spade URL fetched at startup.
- The drops.Watcher (in internal/drops/watch.go) is unchanged — it
  still uses gql.SendMinuteWatched for drop credit. SpadeTracker
  skips whatever channel the Watcher owns, so there's no double-
  heartbeat on the drop channel.
- gql.SendMinuteWatched stays in place; it's still the right call for
  drop credit and may be useful in future for non-Spade flows.

Architecture after this fix (clean separation):
  drops.Watcher    → 1 channel,   sendSpadeEvents GQL,   drop credit
  twitch.SpadeTracker → 2 channels, POST spade.twitch,  channel-points

== 2026-04-07 01:12 == Feature: Detect already-completed drop campaigns via gameEventDrops (v1.4.0) ==

Fixed a critical issue where already-completed drop campaigns (e.g. GZW Stage 2) showed
as 0% and were re-farmed unnecessarily. Root cause: Twitch removes fully-claimed campaigns
from dropCampaignsInProgress (Inventory), and the Dashboard returns no progress data
(self is always null). Without either source providing claimed status, completed campaigns
looked identical to brand-new ones.

Solution (same approach as DevilXD's TwitchDropsMiner):
The Inventory API also returns `gameEventDrops` — a permanent list of ALL ever-claimed
benefit IDs with `lastAwardedAt` timestamps. This persists even after campaigns are removed
from dropCampaignsInProgress. By cross-referencing each drop's benefit ID against this list
(with a campaign time window check to avoid false matches across campaigns), we can now
correctly identify already-completed drops.

4-step merge pipeline:
1. Dashboard → all campaigns (no progress data)
2. Inventory → in-progress campaigns + gameEventDrops (all claimed benefit history)
3. Merge inventory progress into dashboard campaigns (by drop ID)
4. For drops NOT in inventory: check benefit ID against gameEventDrops with timestamp
   validation → mark as claimed if matched within campaign time window

The CompletedCampaigns config cache (from v1.3.9) remains as a second layer — once a
campaign is detected as fully claimed, its ID is stored in config.json so future cycles
skip it immediately without needing the API check.

- Added BenefitID field to TimeBasedDrop (parsed from benefitEdges[0].benefit.id)

- New isWithinCampaignWindow() validates award timestamps against campaign period
- GetDropsInventory() merge step 4: cross-reference benefits with gameEventDrops

== 2026-02-13 02:53 == Feature: Twitch Drops Mining (v1.2.0-beta) ==

What are Twitch Drops?
  Twitch Drops are game-specific rewards (in-game items, skins, emotes, etc.) that
  players earn by watching eligible streams for a required amount of time. Normally
  you must manually find a drops-enabled stream, watch it long enough, and then
  click "Claim" on twitch.tv. TwitchPoint now automates this entire process.

How it works:
  1. Every 10 minutes, TwitchPoint queries your Twitch inventory for active drop campaigns
  2. For each campaign, it matches the allowed channels (or game category) against
     your configured channel list — it will NOT add unknown channels
  3. Matched channels are auto-promoted to P0 priority (highest), guaranteeing them
     a Spade heartbeat slot so watch minutes accumulate
  4. When a drop reaches 100% completion, it is automatically claimed via GQL mutation
  5. Once claimed (or the campaign ends), P0 is removed and the channel reverts to
     its normal priority (P1/P2)

Prerequisites:
  - drops_enabled: true in config (default)
  - Your Twitch account must be linked to the game account on the publisher's website
    (e.g., link Riot account for Valorant drops, link Steam for CS2 drops)
  - The channels streaming the drop-enabled game must be in your configured channel list

Priority system:
  P0 (drop active, auto-assigned) → P1 (always watch) → P2 (rotate every 5 min)
  P0 is temporary and fully automatic — no user action needed.

Changes:
- GQL: Inventory query to fetch active campaigns with progress per drop
- GQL: DropsPage_ClaimDropRewards mutation to auto-claim completed drops
- Farmer: dropCheckLoop() — 30s initial delay, then polls every 10 min

  to acquire the same mutex again (Go mutexes are not reentrant)
- Root cause: Close() locked c.mu, then called c.send("QUIT") which also locks c.mu → deadlock
- Fix: extract conn/writer references under lock, release lock, then send QUIT and close outside
- This caused the TUI to freeze when pressing 'q' or Ctrl+C, requiring kill -9 to terminate
- Pre-existing bug, not caused by drops changes
- Files modified: internal/twitch/irc.go
